## Context

Workflow pipelines are expressed as pipe-separated command strings generated by the agent during tool calls, such as:

```
exec --shell 'inbox list --json' | exec --stdin json --shell 'inbox categorize --json' | approve --prompt 'Apply changes?'
```

These strings need to be parsed into the typed `Pipeline` and `Step` structs from `workflow-types-and-envelope`. The parser is the bridge between the agent's text-based pipeline expressions and the typed runtime.

The project is an Elixir AI agent built on the Jido framework. The parser will live at `Goodwizard.Workflow.PipelineParser` and depend only on the Step and Pipeline structs.

## Goals / Non-Goals

**Goals:**

- Parse pipe-separated command strings into a `Pipeline` struct containing an ordered list of `Step` structs
- Handle single-quoted and double-quoted values so embedded shell commands with spaces parse correctly
- Recognize three step types: `exec` (shell command), `approve` (approval gate), and `openclaw.invoke` (tool invocation)
- Parse all supported flags: `--shell`, `--stdin`, `--prompt`, `--preview-from-stdin`, `--limit`, `--each`, `--item-key`, `--args-json`
- Return clear error tuples for malformed input
- Provide a single public entry point: `PipelineParser.parse/1`

**Non-Goals:**

- Full programming language or expression evaluation
- Workflow file parsing from YAML (separate change: `workflow-file-parser`)
- Execution of parsed pipelines (separate change: `workflow-step-runner`)
- Validation of flag values beyond structural correctness
- Streaming or incremental parsing

## Decisions

### 1. Two-phase parse: tokenize then structure

Phase one is a character-level tokenizer that splits the input into tokens (words, quoted strings, flags, pipe delimiters) while respecting quote boundaries. Phase two consumes the token stream and builds Step structs grouped by pipe boundaries.

**Alternative considered**: Regex-based splitting on `|`. Rejected because pipes inside quoted strings would require fragile lookahead patterns.

### 2. Quote handling preserves content, strips delimiters

Both single (`'...'`) and double (`"..."`) quotes are supported. The tokenizer strips delimiters and emits inner content as a single token. Quotes do not nest. This matches the simplicity constraint for AI-generated strings.

**Alternative considered**: Backslash-escaped quotes. Rejected — adds complexity with no current use case.

### 3. Step type is the first token after a pipe boundary

Each segment begins with a step type keyword. Unknown types produce an error tuple rather than silently defaulting.

### 4. Flags parsed as `--key value` pairs

GNU-style `--key value` convention. Boolean flags (`--preview-from-stdin`, `--each`) are standalone. The parser distinguishes them by maintaining an explicit list of boolean flag names.

### 5. Pure function, no side effects

`PipelineParser.parse/1` is a pure function — no GenServer, no filesystem, no process messages.

## Risks / Trade-offs

- **Quote limitation**: No escaped quotes within same-type delimiters. Use the other quote type as outer delimiter if needed.
- **No positional arguments**: `exec ls` fails — must be `exec --shell 'ls'`. Intentional for self-documenting flags.
- **Fixed step type list**: Adding new types requires code change. Acceptable while vocabulary is small.
- **No multi-line input**: Single-line strings only. Matches agent tool call argument format.
