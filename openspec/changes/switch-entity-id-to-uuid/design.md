## Context

Brain entity IDs are generated by `Goodwizard.Brain.Id` using the `sqids` library. The module maintains a monotonic counter in `brain/.counter`, uses file locking with exponential backoff for concurrency, and includes counter recovery logic that scans existing entity files. The ID pattern is `^[a-z0-9]{8,64}$`.

Cross-entity references in schemas use patterns like `^companies/[a-z0-9]{8,64}$` (typed) and `^[a-z_]+/[a-z0-9]{8,64}$` (polymorphic). These patterns are defined in `Seeds.entity_ref/1`, `Seeds.entity_ref_list/1`, and inline in the `notes` schema.

The `Paths` module exposes `counter_path/1` for the counter file location.

## Goals / Non-Goals

**Goals:**
- Replace sqids ID generation with UUID v4 (`UUID.uuid4()` from Elixir stdlib — no external dependency needed since Elixir ships with `:crypto`)
- Simplify `Goodwizard.Brain.Id` to just generation and validation (remove counter, locking, recovery)
- Update all schema patterns to match UUID format
- Remove the `sqids` hex dependency

**Non-Goals:**
- No migration of existing sqids-based entities — this is a clean break
- No backward-compatible dual-format validation
- No changes to CRUD logic in `Goodwizard.Brain` (only the ID module and schemas change)

## Decisions

**1. Use Elixir's built-in UUID generation**

Elixir provides UUID v4 generation without external dependencies. The implementation is a simple call — no counter file, no locking, no recovery logic needed.

Alternative considered: `elixir_uuid` or `uniq` hex packages — rejected because stdlib is sufficient and avoids adding a dependency to replace one.

**2. UUID pattern: standard 8-4-4-4-12 lowercase hex with hyphens**

The UUID format is `^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$`. This is the canonical UUID string representation.

Entity reference patterns become:
- Typed: `^companies/[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$`
- Polymorphic: `^[a-z_]+/[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$`

**3. Rewrite `Id` module, keep same public API**

The module keeps `generate/1`, `valid?/1`, and `id_pattern/0` — same function signatures, different internals. `generate/1` still takes `workspace` for API compatibility even though it no longer needs it (UUIDs don't require filesystem access). This avoids changing all callers.

**4. Remove counter infrastructure**

Delete all counter-related code from `Id`: `locked_generate`, `read_counter`, `write_counter`, `recover_counter`, `maybe_clean_stale_lock`, `list_entity_ids`, `decode_id`. The `Paths.counter_path/1` function can remain (harmless) or be removed — prefer removing to avoid dead code.

**5. Clean break for existing data**

Existing entity files with sqids-format IDs and filenames won't match the new UUID pattern. This is acceptable for a pre-production system. No migration tooling is built.

## Risks / Trade-offs

**[Longer filenames]** → UUID filenames are 36 chars vs ~8 for sqids. Acceptable — filesystem path limits are not a concern for brain entities.

**[Existing data invalidated]** → Intentional clean break. Existing brain directories should be cleared before using the updated code.

**[`generate/1` still takes workspace param]** → Slight API oddity since UUIDs don't need filesystem access, but avoids a breaking change to callers. The param is simply ignored.
