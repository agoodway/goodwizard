## Context

Goodwizard is an Elixir-based AI agent built on the Jido framework. It uses a workspace directory (`priv/workspace` by default) for file storage, memory, and skills. The agent already has filesystem actions (read, write, edit, list), a memory subsystem (history + long-term), and a prompt skills plugin that discovers `SKILL.md` files.

The current memory system stores unstructured text. There is no way to store, validate, or query structured entities. This change builds the infrastructure layer for a personal knowledge base ("brain") with typed entities backed by JSON Schema validation.

## Goals / Non-Goals

**Goals:**
- Safe, workspace-relative path helpers for the `brain/` directory structure
- Sqids-based short unique ID generation with monotonic counter
- JSON Schema loading, resolution, and validation using `ex_json_schema` (draft 7)
- Ship 6 default JSON Schema definitions: people, places, events, notes, tasks, companies
- Schema seeding on first use

**Non-Goals:**
- Entity CRUD operations (see `brain-entity-storage`)
- Agent-facing Jido actions (see `brain-agent-actions`)
- Full-text search or indexing
- Schema migration tooling (see `brain-schema-migration`)

## Decisions

### 1. Directory structure

```
brain/
  schemas/
    people.json
    places.json
    events.json
    notes.json
    tasks.json
    companies.json
  .counter                <- monotonic counter for Sqids input
  people/
  places/
  events/
  notes/
  tasks/
  companies/
```

**Rationale**: One directory per entity type keeps things organized and scannable. Schema files are JSON (not YAML) because `ex_json_schema` works with JSON/map input natively.

### 2. Module structure

```
Goodwizard.Brain.Paths    — Path helpers (workspace-relative, safe)
Goodwizard.Brain.Id       — Sqids-based ID generation with monotonic counter
Goodwizard.Brain.Schema   — Schema loading, validation, resolution
Goodwizard.Brain.Seeds    — Default schema definitions and seeding
```

### 3. Path safety

`Brain.Paths` rejects path traversal attempts (`..`, leading `/`, null bytes) in entity type names and IDs. All paths are resolved relative to the workspace `brain/` directory.

Path helpers:
- `brain_dir/1` — root brain directory for a workspace
- `schemas_dir/1` — `brain/schemas/` directory
- `entity_type_dir/2` — `brain/<type>/` directory
- `entity_path/3` — `brain/<type>/<id>.md` file path
- `schema_path/2` — `brain/schemas/<type>.json` file path
- `counter_path/1` — `brain/.counter` file path

### 4. Entity ID generation with Sqids

Entity IDs are short, URL-friendly strings generated by the Sqids library (`{:sqids, "~> 0.2.1"}`). Each ID is encoded from a monotonic counter value stored in `brain/.counter`.

Configuration:
- **Alphabet**: lowercase alphanumeric only (`abcdefghijklmnopqrstuvwxyz0123456789`)
- **Minimum length**: 8 characters

ID generation flow:
1. Read current counter from `brain/.counter` (or start at `0` if file doesn't exist)
2. Increment the counter
3. Encode the counter value via `Sqids.encode!(sqids, [counter])`
4. Write the new counter value back to `brain/.counter`
5. Use the encoded string as the entity `id` and filename

```elixir
{:ok, sqids} = Sqids.new(alphabet: "abcdefghijklmnopqrstuvwxyz0123456789", min_length: 8)
id = Sqids.encode!(sqids, [42])  # => "k3g7qae5"
```

**Rationale**: Sqids produce short, human-readable, profanity-free IDs that are easier to work with than UUIDs. The monotonic counter guarantees uniqueness within the workspace. Lowercase-only keeps filenames clean and avoids case-sensitivity issues across filesystems.

### 5. JSON Schema draft 7 with `ex_json_schema`

Each entity type has a JSON Schema file in `brain/schemas/`. The schema defines required fields, types, and constraints. The `ex_json_schema` library resolves schemas once and validates entity data against them.

Each schema includes a top-level `version` integer (starting at `1`) for future migration support (see `brain-schema-migration` change).

Common fields included in every schema:
- `id` (string, required, sqid pattern) — short unique identifier generated by Sqids, used as filename
- `created_at` (string, date-time format)
- `updated_at` (string, date-time format)
- `tags` (array of strings, optional)

Entity reference fields use a `pattern` constraint matching `^[a-z_]+/[a-z0-9]{6,}$` to enforce the `entity-type/sqid` format.

### 6. Default schemas

Six initial entity types ship as embedded Elixir maps in `Brain.Seeds`:

#### people.json

```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "title": "Person",
  "version": 1,
  "type": "object",
  "required": ["id", "name"],
  "properties": {
    "id": { "type": "string", "pattern": "^[a-z0-9]{6,}$" },
    "name": { "type": "string" },
    "email": { "type": "string", "format": "email" },
    "phone": { "type": "string" },
    "company": {
      "type": "string",
      "pattern": "^companies/[a-z0-9]{6,}$",
      "description": "Entity reference to companies"
    },
    "role": { "type": "string" },
    "notes": {
      "type": "array",
      "items": {
        "type": "string",
        "pattern": "^notes/[a-z0-9]{6,}$"
      },
      "description": "Entity references to notes"
    },
    "tags": { "type": "array", "items": { "type": "string" } },
    "created_at": { "type": "string", "format": "date-time" },
    "updated_at": { "type": "string", "format": "date-time" }
  },
  "additionalProperties": false
}
```

## Risks / Trade-offs

- **[YAML parsing edge cases]** — Some YAML values (dates, booleans) get auto-coerced. Mitigation: use quoted strings in frontmatter for ambiguous values; document in action descriptions.
- **[File conflicts]** — No locking mechanism for concurrent counter writes. Mitigation: single-agent usage means no real concurrency risk in practice.
