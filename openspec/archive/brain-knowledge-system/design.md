## Context

Goodwizard is an Elixir-based AI agent built on the Jido framework. It uses a workspace directory (`~/.goodwizard/workspace` by default) for file storage, memory, and skills. The agent already has filesystem actions (read, write, edit, list), a memory subsystem (history + long-term), and a prompt skills plugin that discovers `SKILL.md` files.

The current memory system stores unstructured text. There is no way to store, validate, or query structured entities. The user wants a personal knowledge base ("brain") with typed entities backed by JSON Schema validation.

## Goals / Non-Goals

**Goals:**
- File-based entity storage in `brain/` subdirectory of workspace with YAML frontmatter for structured data
- JSON Schema validation of entity data using `ex_json_schema` (draft 7)
- Ship 6 initial entity types: people, places, events, notes, tasks, companies
- CRUD operations exposed as Jido actions the agent can invoke
- Schema management: create, read, update schemas for entity types
- Support creating entirely new entity types at runtime

**Non-Goals:**
- Full-text search or indexing (can be added later)
- Relationships/graph queries between entities (future enhancement)
- Database or external storage — this is purely file-based
- UI or web interface for brain data
- Schema migration tooling — handled by separate `brain-schema-migration` change

## Decisions

### 1. File format: Markdown with YAML frontmatter

Entity files use standard markdown with YAML frontmatter for structured fields:

```markdown
---
id: "k3g7qae5"
name: John Doe
email: john@example.com
company: "companies/x9rku2dq"
tags: [friend, colleague]
created_at: "2026-02-14T00:00:00Z"
updated_at: "2026-02-14T00:00:00Z"
---

Free-form notes about John go here.
Met at the conference in Austin.
```

**Rationale**: Markdown + YAML frontmatter is human-readable, git-friendly, and widely supported. The `yaml_elixir` dependency already exists in the project. Structured fields live in frontmatter (validated by JSON Schema), while the body holds freeform notes.

**Alternative considered**: Pure JSON files — rejected because they're less human-readable and don't support freeform notes naturally.

### 2. Directory structure

```
brain/
  schemas/
    people.json
    places.json
    events.json
    notes.json
    tasks.json
    companies.json
  .counter                ← monotonic counter for Sqids input
  people/
    k3g7qae5.md
    f1cspw92.md
  places/
    bg6sp8yz.md
  events/
    ...
  notes/
    ...
  tasks/
    ...
  companies/
    x9rku2dq.md
```

**Rationale**: One directory per entity type keeps things organized and scannable. Schema files are JSON (not YAML) because `ex_json_schema` works with JSON/map input natively. Entity filenames use the entity `id` field (a Sqid).

### 3. JSON Schema draft 7 with `ex_json_schema`

Each entity type has a JSON Schema file in `brain/schemas/`. The schema defines required fields, types, and constraints. The `ex_json_schema` library resolves schemas once and validates entity data against them.

Each schema includes a top-level `version` integer (starting at `1`) for future migration support (see `brain-schema-migration` change).

Common fields included in every schema:
- `id` (string, required, sqid pattern) — short unique identifier generated by Sqids, used as filename
- `created_at` (string, date-time format)
- `updated_at` (string, date-time format)
- `tags` (array of strings, optional)

Entity reference fields use a `pattern` constraint matching `^[a-z_]+/[a-z0-9]{6,}$` to enforce the `entity-type/sqid` format.

**Rationale**: JSON Schema is a well-known standard. `ex_json_schema` is the mature Elixir library for it. Draft 7 is the latest version supported and offers good feature coverage.


**Rationale**: Follows existing patterns — thin action modules that delegate to a domain module (like filesystem actions delegate to `Filesystem.resolve_path`). The `Brain` module is the core logic; actions are the agent-facing interface.

### 5. Entity references use `entity-type/sqid` format

Fields that reference another entity use the string format `<entity_type>/<sqid>` — e.g., `"companies/x9rku2dq"`. This encodes both the type and identity of the target in a single value that can be split on `/` to resolve.

Reference fields in the initial schemas:
- People: `company` → `companies/<sqid>`, `notes` → array of `notes/<sqid>`
- Places: `notes` → array of `notes/<sqid>`
- Events: `location` → `places/<sqid>`, `attendees` → array of `people/<sqid>`, `notes` → array of `notes/<sqid>`
- Notes: `related_to` → array of `<entity_type>/<sqid>` (polymorphic)
- Tasks: `assignee` → `people/<sqid>`, `notes` → array of `notes/<sqid>`
- Companies: `contacts` → array of `people/<sqid>`, `notes` → array of `notes/<sqid>`

The `notes` ↔ `related_to` fields form a many-to-many relationship: any entity can link to multiple notes via its `notes` array, and any note can link back to multiple entities of any type via its `related_to` array.

**Rationale**: Embedding the entity type in the reference makes references self-describing — you can resolve them without knowing which field they came from. The `/` separator mirrors the filesystem path (`brain/companies/<sqid>.md`), keeping the mental model consistent.

### 6. Entity ID generation with Sqids

Entity IDs are short, URL-friendly strings generated by the [Sqids](https://sqids.org/elixir) library (`{:sqids, "~> 0.2.1"}`). Each ID is encoded from a monotonic counter value stored in `brain/.counter`.

Configuration:
- **Alphabet**: lowercase alphanumeric only (`abcdefghijklmnopqrstuvwxyz0123456789`)
- **Minimum length**: 8 characters

ID generation flow:
1. Read current counter from `brain/.counter` (or start at `0` if file doesn't exist)
2. Increment the counter
3. Encode the counter value via `Sqids.encode!(sqids, [counter])`
4. Write the new counter value back to `brain/.counter`
5. Use the encoded string as the entity `id` and filename

```elixir
{:ok, sqids} = Sqids.new(alphabet: "abcdefghijklmnopqrstuvwxyz0123456789", min_length: 8)
id = Sqids.encode!(sqids, [42])  # => "k3g7qae5"
```

If an entity with the same ID exists, the operation returns an error (create) or overwrites (update).

**Rationale**: Sqids produce short, human-readable, profanity-free IDs that are easier to work with than UUIDs. The monotonic counter guarantees uniqueness within the workspace. Lowercase-only keeps filenames clean and avoids case-sensitivity issues across filesystems.

## Risks / Trade-offs

- **[No search/indexing]** → Listing entities requires directory scan. Acceptable for hundreds of entities; would need indexing for thousands. Mitigation: can add search later.
- **[No referential integrity]** → Entity references (e.g., `"companies/x9rku2dq"`) are `entity-type/sqid` strings with no enforcement that the target exists. Mitigation: a future "validate refs" command could check that referenced entities exist.
- **[Schema evolution]** → Updating a schema does not retroactively validate or migrate existing entities. Mitigation: the separate `brain-schema-migration` change adds versioning and migration support.
- **[File conflicts]** → No locking mechanism for concurrent writes. Mitigation: single-agent usage means no real concurrency risk in practice.
- **[YAML parsing edge cases]** → Some YAML values (dates, booleans) get auto-coerced. Mitigation: use quoted strings in frontmatter for ambiguous values; document in action descriptions.
